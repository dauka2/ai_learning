## 1. Общая архитектура

Система состоит из трёх слоёв:

**Бэкенд (Python)** — FastAPI-сервер (`app.py`). Принимает HTTP-запросы от браузера, управляет сессиями, вызывает ИИ-модели, читает и пишет в базу данных. Всё в одном процессе, без микросервисов.

**Фронтенд (JavaScript)** — одностраничное приложение (`static/app.js` + `static/index.html`). Работает целиком в браузере, обменивается с бэкендом через JSON API. Никаких фреймворков — чистый JS.

**Хранилище** — SQLite-файл `data/app.db` через SQLAlchemy ORM (`db.py`). Хранит пользователей, сессии, ответы, состояния навыков. Банк вопросов хранится в памяти (не в БД), пересобирается при запуске из `content.py`.

**Внешние зависимости:**
- `Groq API` (llama-3.3-70b-versatile) — генерация новых вопросов и анализ ошибок. Опционально: без ключа система работает на куратированном банке.
- `model.pkl` — обученная нейросеть (numpy-вычисления, без PyTorch/TensorFlow).
- `recommender.pkl` — обученный Random Forest (scikit-learn).

## 2. CSV-файлы

Все пять CSV-файлов использовались **только при обучении моделей**. В работающем сервере они не нужны — модели уже обучены и сохранены в `.pkl`. Файлы оставлены для воспроизводимости эксперимента.

### 1.csv — 100 000 студентов, 43 признака (LearnerProfileNet)

Это основной датасет для обучения нейросети `model.pkl`. Каждая строка — один студент с историей онлайн-обучения.

Ключевые столбцы которые используются в модели:
- `digital_literacy_score` — цифровая грамотность (1–10)
- `skill_pre_score` — начальный уровень знаний (0–100)
- `video_completion_pct` — процент просмотренных видеолекций
- `assignment_submission_rate` — процент сданных заданий
- `content_difficulty_avg` — средняя сложность контента, который выбирал студент (1–5)
- `engagement_consistency` — регулярность занятий (0–1)
- `total_learning_hours` — суммарные часы обучения
- `in_app_quiz_score` — средний балл за квизы в приложении
- `session_count_weekly` — количество сессий в неделю

Целевая переменная: `mastery_score` — итоговый балл освоения материала. Нейросеть учится предсказывать его по 9 признакам выше.

Остальные 34 столбца (эссе, MOOC-платформа, страна, гендер и т.д.) в модель не вошли — их корреляция с `mastery_score` оказалась слабее.

### 2.csv — 1 000 студентов, 18 признаков (recommender.pkl, часть)

Поведенческие характеристики студентов. Использовался для обучения части Random Forest, предсказывающей пробелы в знаниях.

Признаки, которые попали в `recommender.pkl`:
- `daily_study_hours` — часов учёбы в день
- `attendance_percentage` — посещаемость
- `homework_completion_rate` — выполнение домашних заданий
- `sleep_hours` — часов сна
- `screen_time_hours` — экранное время
- `physical_activity_minutes` — физическая активность
- `motivation_score` — мотивация (0–10)
- `exam_anxiety_score` — тревожность на экзаменах

Целевые переменные: `final_score` и `pass_fail` по трём доменам (Математика, Естественные науки, Языки).

### 3.csv — 9 777 записей по неделям (вспомогательный)

Понедельная динамика 977 студентов (10 недель на каждого). Содержит `quiz_score`, `assignment_score`, `performance_index`, `fatigue_index`. Использовался для анализа трендов и валидации того, что `engagement_consistency` из 1.csv действительно коррелирует с финальным результатом.

### 4.csv — 6 608 студентов, факторы успеваемости (вспомогательный)

Широкий датасет с качественными переменными: `Parental_Involvement` (Low/Medium/High), `Access_to_Resources`, `School_Type` (Public/Private), `Peer_Influence`, `Learning_Disabilities`. Целевая переменная `Exam_Score`.

Использовался при разработке доменной структуры рекомендаций — помог определить, что «Родительская вовлечённость» и «Доступ к ресурсам» существенно влияют на результат в разных предметных доменах, поэтому рекомендации делятся на три домена, а не по одному предмету.

### 5.csv — 296 вопросов по Информатике (банк вопросов)

Реальные вопросы с типом MCQ (multiple choice question) из экзаменационных тестов 2015–2022 годов. Формат колонки `question`:

```
Текст вопроса | Вариант A | Вариант B | Вариант C | Вариант D
```

Правильный ответ не размечен в файле — он определяется по полю `type` и году, либо первый вариант считается верным при загрузке. Функция `load_cs_questions()` в `content.py` парсит этот файл и добавляет вопросы в банк с уровнем сложности в зависимости от года (`year_to_difficulty`).

## 3. model.pkl — нейросеть LearnerProfileNet

### Что это

Нейронная сеть прямого распространения (feedforward), обученная предсказывать вероятность освоения материала P(mastery) по 9 поведенческим признакам нового студента. Используется один раз при регистрации, чтобы определить стартовый уровень θ.

### Архитектура: 9 → 64 → 32 → 1

- **Входной слой**: 9 нейронов (по числу признаков)
- **Скрытый слой 1**: 64 нейрона, активация ReLU
- **Скрытый слой 2**: 32 нейрона, активация ReLU
- **Выходной слой**: 1 нейрон, активация Sigmoid → число от 0 до 1

### Что хранится в файле

```
model.pkl — Python dict:
  W1: матрица весов (9×64)
  b1: вектор смещений (64,)
  W2: матрица весов (64×32)
  b2: вектор смещений (32,)
  W3: матрица весов (32×1)
  b3: вектор смещений (1,)
  features: список из 9 названий признаков
  labels_ru: русские названия признаков
  ranges: min/max/default для каждого признака
  mean, std: параметры стандартизации обучающей выборки
  arch: "9-64-32-1"
  epochs: 80
  val_acc: 0.739
  val_auc: 0.619
  threshold: 55
  positive_rate: 0.26
```

### Метрики

- `val_acc = 0.739` — точность на валидационной выборке 73.9%
- `val_auc = 0.619` — ROC-AUC 0.619 (заметно лучше случайного 0.5)
- `positive_rate = 0.26` — 26% студентов в датасете имеют высокое освоение
- Baseline (всегда предсказывать «не освоит») дал бы точность 74%, поэтому AUC важнее accuracy

### На чём обучали

Датасет `1.csv`, 100 000 студентов. Признаки стандартизированы (вычтено среднее, делено на σ). Целевая переменная — бинаризованный `mastery_score` (порог 55 из 100). Обучение: 80 эпох, реализовано вручную на NumPy (backpropagation без фреймворков).

### Как используется в системе

Функция `lpn_predict_p_mastery()` в `ai_core.py`:
1. Берёт словарь из 9 признаков студента
2. Нормализует их: `x = (x - mean) / std`
3. Прогоняет через три слоя: `a1 = ReLU(x @ W1 + b1)`, `a2 = ReLU(a1 @ W2 + b2)`, `p = sigmoid(a2 @ W3 + b3)`
4. Возвращает P(mastery) от 0 до 1

Функция `p_mastery_to_theta()` переводит P в стартовый θ по формуле:
```
θ = clip((P - 0.26) * 8.0, -1.5, 1.5)
```
- P=0.26 (среднее) → θ=0.0 (средний уровень)
- P=0.0 → θ≈-2.1 (начинающий)
- P=1.0 → θ≈+1.5 (продвинутый)

### Когда модель не работает

Если `model.pkl` отсутствует — `load_learner_profile_net()` возвращает None, стартовый θ=0.0, оценка пропускается. Система продолжает работать.

## 4. recommender.pkl — Random Forest

### Что это

Мультивыходный классификатор (Multi-Output RandomForestClassifier из scikit-learn). Предсказывает, есть ли у студента пробел в знаниях в каждом из трёх учебных доменов.

### Три домена

- **Математика** — включает предметы: Математика, Информатика
- **Естественные науки** — Физика, Химия, Биология
- **Языки** — История, Английский язык

### Входные признаки (8 штук)

Поведенческие характеристики из `2.csv`:
- `daily_study_hours`, `attendance_percentage`, `homework_completion_rate`
- `sleep_hours`, `screen_time_hours`, `physical_activity_minutes`
- `motivation_score`, `exam_anxiety_score`

### Что хранится в файле

```
recommender.pkl — Python dict:
  model: MultiOutputClassifier(RandomForestClassifier)
  features: список 8 признаков
  labels_ru: русские названия
  ranges: min/max/default для каждого признака
  medians: медианы для подстановки при отсутствии данных
  domains: ['Математика', 'Естественные науки', 'Языки']
  subject_to_domain: маппинг предмета → домен
  auc: {'Математика': 0.884, 'Естественные науки': 0.922, 'Языки': 0.887}
  gap_threshold: порог вероятности пробела
  model_name: "Multi-Output RandomForest"
  n_estimators: число деревьев
  train_size: размер обучающей выборки
```

### Метрики

ROC-AUC по доменам: 0.884 / 0.922 / 0.887 — высокие значения, модель надёжно разделяет студентов с пробелами от студентов без них.

### Как используется

`RecommendationModel.predict_gaps(profile)` в `ai_core.py`:
1. Берёт словарь поведенческих характеристик студента
2. Если признак отсутствует — подставляет медиану из обучающей выборки
3. Вызывает `model.predict_proba()` — Random Forest возвращает вероятности по каждому дереву
4. Возвращает: `{'Математика': 0.73, 'Естественные науки': 0.21, 'Языки': 0.65}`

Результат используется в `recommend.py` при построении плана — если вероятность пробела высокая, этот домен попадает в приоритет рекомендаций.

## 5. ai_core.py

Ядро всей адаптивной логики. Здесь живут IRT-модель, политика подбора вопросов, и функции работы с нейросетями.

### Константы сложности

```python
B_BY_DIFFICULTY = {
    "primary": -2.0,  # 1–4 класс
    "easy":    -1.0,  # 5–6 класс
    "medium":   0.0,  # 7–8 класс
    "hard":    +1.0,  # 9–10 класс
    "expert":  +2.0,  # 11 класс
}
```

`b` — параметр сложности задания в теории IRT (Item Response Theory). Чем выше b, тем сложнее вопрос.

### STAGES — ступени обучения

Каждая ступень определяет:
- `label` — название для UI
- `difficulties` — допустимые уровни сложности вопросов
- `start_theta` — начальный θ при выборе этой ступени
- `fixed` — всегда False (адаптация всегда включена)

Ступени `grade_1`–`grade_11` дают каждому классу свой диапазон. Ступени `primary`, `easy`, `medium`, `hard`, `expert` — диапазонные псевдонимы, скрытые от пользователя в UI. Ступень `auto` — весь диапазон без ограничений.

### IRTSkillTracker — отслеживание уровня θ

Это реализация **однопараметрической модели Раша** (Rasch 1PL, частный случай IRT).

**Ключевая идея IRT**: вероятность правильного ответа зависит от разницы между способностью студента (θ) и сложностью вопроса (b):

```
P(верно | θ, b) = 1 / (1 + e^(-(θ - b)))
```

- Если θ >> b: студент сильнее задания, ответит верно с высокой вероятностью
- Если θ << b: задание сильнее студента, скорее ответит неверно
- Если θ = b: вероятность 50/50

**Обновление θ после каждого ответа** (градиентный шаг):
```
θ_new = θ + lr * (outcome - P(θ, b))
```
- `outcome` = 1.0 (верно) или 0.0 (неверно)
- `P(θ, b)` = предсказанная вероятность до ответа
- `lr` = скорость обучения, убывает с числом попыток: `lr = max(0.08, 0.60 / (1 + 0.15 * attempts))`

Смысл: если студент ответил верно там, где ожидалась ошибка (outcome > P), θ растёт сильнее.

**Серия правильных ответов**: при `streak >= 3` скорость обучения умножается на `STREAK_BONUS = 1.3` — θ растёт быстрее когда студент явно в потоке.

**wrong_streak**: счётчик неверных ответов подряд. При `wrong_streak >= 2` политика снижает сложность.

**Стандартная ошибка SE(θ)**: накапливается через информацию Фишера `I = P*(1-P)`. `SE = 1/√(∑I)`. Чем больше ответов, тем точнее оценка θ, тем меньше SE. Используется как критерий остановки теста.

**Методы:**

`prob_correct(difficulty)` — вероятность правильного ответа при текущем θ и заданной сложности.

`optimal_difficulty(allowed)` — возвращает уровень сложности, ближайший к `θ - 0.3`. Смещение -0.3 означает что идеальный вопрос чуть легче текущего θ — студент в зоне ближайшего развития, P(верно) ≈ 57%.

`level_label()` — текстовый уровень: Начинающий (θ < -1), Базовый (-1 до -0.2), Средний (-0.2 до 0.6), Продвинутый (0.6 до 1.5), Мастер (> 1.5).

`normalized_level()` — θ от 0 до 1 (линейно из диапазона [-3, 3]).

### IRTSkillManager

Контейнер трекеров. Хранит один `IRTSkillTracker` на каждую пару (предмет, тема). Методы:

`tracker(subject, topic, theta0)` — возвращает существующий трекер или создаёт новый.

`update(subject, topic, is_correct, difficulty)` — обновляет θ по ответу.

`weakest_topic_in_subject(subject)` — находит тему с минимальным θ среди тех, по которым есть хотя бы один ответ.

`subject_theta(subject)` — взвешенное среднее θ по всем темам предмета (вес = число попыток).

### AdaptivePolicyEngine — политика подбора

Принимает решение о следующем вопросе: какой уровень сложности и нужно ли перефразировать.

**Иерархия приоритетов (выполняется первый подходящий):**

1. **REINFORCE + rephrase**: последний ответ неверен И детектор нашёл конкретную концептуальную ошибку. Снижаем сложность на шаг, просим Groq сгенерировать новый вопрос (rephrase=True).

2. **REINFORCE**: серия из 2+ неверных ответов подряд (wrong_streak ≥ 2). Снижаем сложность на шаг без перефразирования. Это помогает при затруднении даже без Groq.

3. **PROGRESS / CHALLENGE**: серия из 3+ верных ответов (streak ≥ 3). Повышаем сложность. Если θ > 1.0, действие называется CHALLENGE, иначе PROGRESS (только метка для UI).

4. **MAINTAIN**: стандартный случай. Задаём вопрос оптимальной сложности (θ - 0.3).

`step_difficulty(difficulty, delta, allowed)` — сдвигает уровень на ±1 позицию в пределах допустимых уровней ступени.

### Функции нейросети

`load_learner_profile_net(path)` — загружает model.pkl.

`lpn_predict_p_mastery(artifact, features)` — прогон через нейросеть (NumPy):
```
x = (features - mean) / std
a1 = ReLU(x @ W1 + b1)
a2 = ReLU(a1 @ W2 + b2)
p  = sigmoid(a2 @ W3 + b3)
```

`lpn_start_theta(artifact, features)` — возвращает (P, θ).

`p_mastery_to_theta(p)` — `clip((p - 0.26) * 8.0, -1.5, 1.5)`.

### LPN_PRESETS

Три предустановленных профиля для выбора на экране онбординга:

- **Начинающий**: низкая грамотность (3.5), низкий предыдущий балл (15), мало часов (2/нед)
- **Средний**: средние значения из датасета
- **Сильный**: высокая грамотность (9.0), балл 75, 12 сессий/неделю

При выборе профиля его числа напрямую идут в нейросеть, минуя ползунки.

## 6. content.py

Весь учебный контент и генерация вопросов через Groq.

### SUBJECTS — предметная таксономия

Словарь `{предмет: [список тем]}`. Семь предметов, 2–6 тем каждый:
- Математика (4 темы), Физика (4), Химия (3), Биология (3), Информатика (5), Английский язык (3), История (3)

### SUBJECTS_BY_GRADE — предметы по классам

Словарь `{класс: [список предметов]}` для классов 1–11:
- Классы 1–4: Математика, История, Английский язык
- Классы 5–6: добавляется Биология
- Классы 7–9: добавляются Физика и Химия
- Классы 10–11: добавляется Информатика

`subjects_for_grade(grade)` — возвращает список предметов для класса.

`subjects_for_stage(stage)` — конвертирует ID ступени в номер класса через `_STAGE_TO_GRADE` и вызывает `subjects_for_grade`.

### CONCEPTS — текстовые концепции тем

Словарь `{(предмет, тема): строка}`. Краткое определение ключевого понятия темы. Раньше показывалось после каждого ответа, сейчас убрано из UI (Проблема 9), но данные остались в структуре.

### _BANK — куратированный банк вопросов

Словарь `{(предмет, тема): список вопросов}`. Каждый вопрос — кортеж:
```python
(difficulty, question_text, [option_a, option_b, option_c, option_d], correct_letter, explanation, common_mistake)
```

Покрывает все 5 уровней сложности для основных тем. Например, для Математики/Арифметики: задача про 7+6 (primary), процент от числа (easy/medium), сложный процент (expert).

`curated_cards()` — превращает банк в список объектов `QuestionCard`.

### Парсинг 5.csv для информатики

`load_cs_questions()` читает файл `5.csv`. Каждая строка — вопрос с вариантами через `|`. Функция `year_to_difficulty()` назначает сложность по году: до 2017 → primary/easy, 2018–2020 → medium, после → hard/expert. `cs_topic_of()` определяет тему по ключевым словам в тексте вопроса.

**Важный момент**: в 5.csv нет разметки правильного ответа. При загрузке `correct_answer=None`. Такие вопросы не попадают в активный банк через `fetch_questions`, так как фильтр `correct_answer is not None` не пройдут. Они фактически не используются без дополнительной разметки.

### QuestionCard — структура вопроса

Pydantic-модель:
```python
question: str          # текст вопроса
options: list[str]     # 4 варианта (тексты, без букв)
correct_answer: str    # буква: "a", "b", "c" или "d"
explanation: str       # объяснение правильного ответа
common_mistake: str    # типичная ошибка
concept: str           # концепция темы
subject: str
topic: str
difficulty: str
source: str            # "curated", "llm", "dataset_5csv"
card_id: str           # уникальный ID
```

### generate_question() — генерация через Groq

Функция генерирует один вопрос через LLM при условии:
- передан `groq_client`
- в банке нет вопросов нужной сложности (или нужно перефразировать)

**Системный промпт** (ключевое правило): `options[0]` ВСЕГДА правильный ответ. LLM обязан сначала вычислить ответ, поместить его первым, затем придумать три дистрактора. Поле `correct` в JSON — всегда `"a"`.

**Почему так**: LLM часто ошибался в маппинге «какая буква соответствует правильному ответу» (галлюцинация). Теперь у LLM одна задача — правильно вычислить ответ и поставить его первым. Маппинг букв делает `_shuffle_options` в `app.py`.

**Инструкция для Английского языка**: вопрос на русском, варианты на английском. Для остальных предметов — всё на русском.

**Блок avoid_block**: если передан список `recent_questions` (последние вопросы по теме в сессии), они добавляются в промпт с запретом повторять числа, операции и тип ситуации. Это решает проблему «15−3 карандашей» с разными именами.

**После парсинга**: `correct` всегда принудительно ставится `"a"`, поле от LLM игнорируется. Сгенерированный вопрос сохраняется в банк через `upsert_question` — при следующем запросе той же темы/сложности он берётся из кэша без нового обращения к Groq.

## 7. app.py

FastAPI-сервер. Все HTTP-маршруты, вся оркестрация компонентов.

### Глобальные объекты при старте

```python
conn          # соединение с БД (SQLAlchemy)
LPN           # нейросеть из model.pkl (или None)
RECOMMENDER   # Random Forest из recommender.pkl (или None)
groq_client   # клиент Groq API (или None)
DETECTOR      # MisconceptionDetector
ACTIVE        # dict: session_id → {mgr, current, last_correct, last_misc, stage}
```

`ACTIVE` — in-memory словарь активных сессий. Хранит всё состояние сессии между запросами: текущий трекер θ, текущую карточку, последний ответ. При перезапуске сервера состояние теряется, но θ персистирован в БД через `skill_states`.

### Аутентификация и пользователи

`POST /register` — регистрация. Создаёт пользователя с PBKDF2-хэшем пароля, выдаёт токен.

`POST /login` — логин. Проверяет хэш, возвращает токен.

`GET /onboarding` — данные для экрана онбординга: список ступеней (`STAGES`), пресеты персон (`LPN_PRESETS`), метаданные моделей, список предметов. Фронтенд использует это для построения экрана выбора профиля.

`POST /users/{user_id}/profile` — сохранение профиля после онбординга. Принимает `features` (9 признаков для LPN) и `behavior` (8 поведенческих для RF). Вызывает `_assess_and_save()`.

`_assess_and_save(uid, features, behavior)`:
1. Если есть features и LPN — вызывает `lpn_start_theta()`, получает P и θ
2. Сохраняет в БД через `db.save_profile()`
3. Возвращает `assessment` (P, θ, уровень) или None

### Предметы и темы

`GET /subjects?user_id=...&stage=...` — список предметов с прогрессом пользователя. Если передан `stage`, фильтрует через `subjects_for_stage(stage)` — именно здесь решается Проблема 1 (физика не показывается в 4 классе). Для каждого предмета считается `mastery_pct` из истории ответов.

`GET /topics?subject=...&user_id=...` — список тем предмета с θ по каждой.

### Сессии

`POST /sessions` — создаёт новую учебную сессию. Параметры: `scope` (all/subject/topic), `subject`, `topic`, `stage`. Создаёт запись в БД, инициализирует `ACTIVE[session_id]`.

`_ensure_active(session_id)` — если сессия есть в `ACTIVE`, возвращает её. Если нет (после перезапуска) — восстанавливает из БД: загружает `skill_states`, создаёт трекеры с сохранёнными θ.

### Получение вопроса

`GET /sessions/{session_id}/next` — главный эндпоинт игрового цикла.

Порядок работы:
1. Получить сессию и её `stage`
2. Определить (subject, topic) через `_next_topic()`
3. Получить или создать `IRTSkillTracker` для этой темы
4. Если у пользователя есть сохранённое состояние навыка в БД — загрузить θ
5. Если сессия началась и есть start_theta от LPN — применить
6. Проверить критерий остановки: `answered >= q_limit` или `SE(θ) < 0.4` (для topic-scope)
7. Вызвать `policy.decide()` → получить difficulty и action
8. Получить тексты предыдущих вопросов темы: `db.session_topic_question_texts()`
9. Вызвать `_pick_question()` → получить карточку
10. Перемешать варианты: `_shuffle_options()`
11. Сохранить карточку в `st["current"]`
12. Вернуть вопрос клиенту (без правильного ответа)

`_next_topic(sess, st)` — логика выбора темы:
- `scope=topic`: всегда одна и та же тема из параметров сессии
- `scope=subject`: по очереди темы предмета, приоритет непройденным
- `scope=all`: все темы всех предметов

`_pick_question(subject, topic, difficulty, used_ids, rephrase, allowed, recent_questions)`:

1. Пытается взять вопрос из банка (`db.fetch_questions`) нужного уровня, исключая уже показанные (`used_ids`)
2. Если нет точного уровня — ищет ближайший в пределах разрешённых ступенью
3. Если есть Groq И (нужно перефразировать ИЛИ банк пуст ИЛИ нет нужного уровня) — генерирует через LLM
4. Fallback: берёт любой вопрос по теме, предпочитая непоказанные

`_shuffle_options(card)` — позиционное перемешивание. Берёт текущий `correct_answer` (буква), находит её позицию (индекс), перемешивает список позиций, пересобирает `options` и вычисляет новый `correct_answer`. Исправляет баг с дублирующимися текстами вариантов.

### Проверка ответа

`POST /sessions/{session_id}/answer` — принимает ответ студента.

1. Достаёт карточку из `st["current"]`
2. Чистит ответ от лишних символов (`a)`, `b.` и т.д.)
3. Сравнивает с `card.correct_answer` — строгое равенство букв
4. Обновляет θ через `st["mgr"].update()`
5. Если неверно — запускает `DETECTOR.detect()` для анализа ошибки
6. Записывает ответ в БД (`db.record_response`)
7. Сохраняет обновлённый `skill_state` в БД
8. Возвращает: `is_correct`, `correct_answer`, текст правильного варианта, объяснение, данные о заблуждении

### Результаты и рекомендации

`GET /sessions/{session_id}/results` — итоги сессии: точность, θ, Δθ, ошибки, категории заблуждений.

`POST /sessions/{session_id}/recommend` — генерирует план обучения через `RecommendationEngine`. Если сессия по предмету — рекомендации только по темам этой сессии.

### Аналитика и прогресс

`GET /users/{user_id}/analytics` — полная аналитика: освоение по предметам и темам, слабые/сильные стороны, динамика θ.

`GET /users/{user_id}/progress` — дашборд прогресса: все сессии, охват тем, общая статистика.

### Административные эндпоинты

`GET/POST /admin/topics` — управление темами (добавление, список).
`POST /admin/generate` — ручная генерация вопросов через LLM для заданной темы и сложности.
`GET/PUT /admin/questions` — просмотр и одобрение LLM-вопросов.

## 8. db.py

Слой персистентности через SQLAlchemy ORM.

### Таблицы БД

**users** — пользователи: логин, хэш пароля (PBKDF2-HMAC-SHA256, 120 000 итераций), соль, роль (student/admin), токен сессии.

**profiles** — учебный профиль: 9 LPN-признаков, 8 поведенческих признаков, P(mastery) от нейросети. JSON-поля.

**sessions** — учебные сессии: scope, предмет/тема, время старта/финиша, статус, лимит вопросов.

**responses** — каждый ответ: session_id, user_id, вопрос, card_id, сложность, ответ студента, is_correct, θ до/после, action (REINFORCE/PROGRESS/...), данные о заблуждении (JSON).

**skill_states** — персистентный θ по (user_id, subject, topic): θ, SE, попытки, верные ответы, история θ (JSON-массив). Обновляется после каждого ответа. Позволяет восстановить состояние после перезапуска.

**plans** — сохранённые планы обучения: JSON с блоками (приоритеты, упражнения, расписание, ресурсы, критерии). Привязаны к user_id и session_id.

**plan_items** — задачи из плана: день, текст задачи, флаг выполнения.

### Банк вопросов в памяти

`conn.questions` — словарь `{card_id: dict}` в памяти процесса. Наполняется при старте из `content.curated_cards()`. LLM-вопросы добавляются через `upsert_question()` в момент генерации. При перезапуске LLM-вопросы теряются.

### Ключевые функции

`fetch_questions(conn, subject, topic, difficulty, exclude_card_ids)` — фильтрует in-memory банк. `exclude_card_ids` — множество card_id уже показанных вопросов в сессии.

`session_used_card_ids(conn, session_id)` — возвращает множество card_id всех вопросов, показанных в сессии (из таблицы responses).

`session_topic_question_texts(conn, session_id, subject, topic, limit=6)` — возвращает тексты последних N вопросов по данной теме в сессии. Используется для передачи в Groq чтобы избежать повторений.

`record_response(...)` — записывает ответ в БД с полными метаданными.

`save_skill_state(...)` — upsert в skill_states: если запись есть — обновляет, нет — создаёт.

`user_skill_states(conn, user_id, subject)` — все сохранённые θ пользователя (опционально фильтрация по предмету).

## 9. analytics.py

Вычисление статистик по данным из БД. Не хранит состояние — только агрегирует.

### Константы

```python
WEAK_THETA = -0.2    # ниже — слабая тема
STRONG_THETA = 0.6   # выше — сильная тема
```

### Функции

`topic_mastery(conn, user_id, subject)` — по каждой теме, по которой есть ответы: θ, уровень, точность, mastery_pct (нормализованный θ 0–100%). Возвращает список, отсортированный по θ по возрастанию.

`subject_mastery(conn, user_id)` — θ предмета = взвешенное среднее θ тем (вес = число попыток). Агрегация: если по математике есть θ(-0.3) по алгебре (10 попыток) и θ(0.5) по геометрии (5 попыток), итоговый θ математики = (-0.3*10 + 0.5*5) / 15 ≈ -0.03.

`weak_and_strong(conn, user_id)` — разделяет темы на слабые (θ < -0.2), средние, сильные (θ ≥ 0.6). Возвращает три топ-3.

`mistake_categories(conn, user_id, session_id)` — разбивает неверные ответы по типам заблуждений (из поля `misconception` в responses). Считает частоту каждого типа.

`theta_trend(conn, user_id, subject, topic)` — временной ряд θ по одной теме: каждый ответ → точка (индекс, θ_after, верно/нет, сложность, время).

`subject_trend(conn, user_id, subject)` — скользящее среднее θ по предмету: после каждого ответа пересчитывает среднее по всем темам предмета.

`session_summary(conn, session_id)` — итоги одной сессии: точность, Δθ (рост за сессию), ряд θ, список ошибок с подсказками.

`progress_dashboard(conn, user_id)` — всё сразу для экрана прогресса: освоение предметов, тем, охват (сколько тем начато из всего), история сессий, общая статистика.

## 10. recommend.py

Генерирует персонализированный план обучения на 2 недели.

### Структура плана (5 блоков)

1. **Приоритетные темы** — слабые темы с θ, уровнем, точностью и пробелом по домену (от RF-модели)
2. **Упражнения** — 2–3 задания на каждую слабую тему, адаптированные к уровню (Начинающий → «повторить теорию», Продвинутый → «объяснить по Фейнману»)
3. **Материалы** — реальные открытые ресурсы (Khan Academy, MIT OpenCourseWare, BBC Learning English) конкретно по теме
4. **Расписание** — 2 дня на каждую тему + дни 7 и 14 для интервального повторения
5. **Критерии достижения** — точность ≥ 75% и θ ≥ текущий+0.5 для каждой темы

### _gather(session_id)

Ключевой метод сбора данных для плана.

**Если передан session_id** — рекомендации строятся ТОЛЬКО по темам текущей сессии. Берёт пары (subject, topic) из ответов сессии, фильтрует `topic_mastery` по ним. Решает Проблему 8: после теста по математике не рекомендует физику.

**Если session_id = None** — берёт всю историю пользователя через `weak_and_strong()`.

Также пробует `predict_gaps()` из RF-модели — если профиль поведения сохранён, добавляет вероятность пробела по домену для каждой темы.

### _intro(g, blocks)

Мотивационное вступление к плану. Формирует детерминированный текст из данных, затем (если есть Groq) просит LLM переписать его «живее». LLM не меняет числа и темы — только стиль. Если Groq недоступен — возвращает исходный детерминированный текст.

### _RESOURCES

Словарь `{предмет: {тема: [(название, URL)]}}` с реальными учебными ресурсами. Khan Academy для большинства предметов, MIT OpenCourseWare для алгоритмов, CS50 для SQL, OSTEP для ОС, BBC для английского. `_default` — запасная ссылка для темы без специфического ресурса.

## 11. misconception.py

Двухуровневый детектор концептуальных ошибок. Вызывается только при неверном ответе.

### Уровень 1 — правила (мгновенно, без API)

`_rule_level(subject, question)` — ищет ключевые слова вопроса в словаре `_RULES`. Например, если в вопросе по математике есть «%» или «процент» → тип «Ошибка с процентами» с готовым объяснением и шагами исправления.

Правила покрывают 7 предметов: Информатика (6 правил), Математика (4), Физика (2), Химия (2), Биология (2), Английский (2), История (1).

Если правило найдено → возвращает его немедленно, LLM не вызывается.

### Уровень 2 — LLM (глубокий анализ)

`_llm_level(subject, topic, question, user_answer, correct_answer)` — вызывается если правило не нашлось.

Отправляет в Groq: предмет, тему, вопрос, ответ студента, правильный ответ. Просит вернуть JSON с `type`, `description`, `hint`, `corrective_steps`.

Если Groq недоступен → вызывает `_generic(subject)` — заготовленный общий ответ по предмету.

### Структура возвращаемого объекта

```json
{
  "type": "Ошибка с процентами",
  "description": "Берётся процент от неверной базы...",
  "hint": "x% от числа N это N·x/100. Уточните базу.",
  "corrective_steps": ["Определите базу", "Переведите в долю", "Умножьте"],
  "level": 1
}
```

`level=1` — найдено правилом, `level=2` — найдено LLM или общий ответ.

Результат сохраняется в таблицу `responses` как JSON-строка. Аналитика агрегирует их по полю `type`.

## 12. static/app.js

Одностраничное приложение (SPA). Никаких фреймворков — чистый JavaScript.

### State — глобальное состояние

```javascript
State = {
  token, userId, userName, role,  // авторизация
  onboarding,                     // данные с /onboarding
  persona,                        // выбранный профиль (Начинающий/Средний/Сильный)
  session,                        // текущая сессия {id, subject, topic, ...}
  lastFinish,                     // id последней завершённой сессии
  examMode,                       // true пока идёт тест
  viewingPast,                    // просмотр архивной сессии
  stage,                          // выбранная ступень (grade_1..grade_11, auto)
}
```

Сохраняется в `localStorage` между перезагрузками страницы через `saveUser()` / `loadUser()`.

### Навигация (show/go)

`show(name)` — показывает один экран из `SCREENS`, скрывает остальные через CSS-класс `.on`.

`go(name, arg)` — роутер: определяет какие действия нужны при переходе (загрузка данных, сброс состояния). Каждый переход выключает `examMode`.

### Экраны

- **auth** — логин/регистрация
- **onboarding** — выбор профиля, имя, кнопка «Начать»
- **catalog** — список предметов с прогрессом
- **topics** — темы предмета
- **quiz** — активный тест
- **results** — итоги теста
- **plan** — план обучения
- **analytics** — детальная аналитика
- **progress** — общий прогресс
- **admin** — панель администратора

### Онбординг

`renderOnboarding(d)` — рендерит карточки персон из `d.presets`. Три карточки (Начинающий/Средний/Сильный).

`pickPersona(name)` — выделяет выбранную карточку, сохраняет в `State.persona`.

`populateStages()` — заполняет выпадающий список ступеней. Показывает только `grade_1`–`grade_11` и `auto` (диапазонные псевдонимы скрыты через `STAGE_ORDER`). Показывает подсказку из `STAGE_HINTS`.

`createUser()` — при нажатии «Начать»:
1. Сохраняет имя
2. Берёт пресет выбранной персоны: `presets[State.persona]`
3. Отправляет на `/users/{id}/profile` с features из пресета
4. Если LPN вернул оценку — показывает карточку с P(mastery) и θ
5. Иначе — сразу переходит в каталог

### Тест (quiz)

`startSession(scope, subject, topic)` — создаёт сессию (`POST /sessions`), сохраняет в `State.session`, включает `State.examMode`, вызывает `nextQuestion()`.

`nextQuestion()` — запрашивает `/sessions/{id}/next`, рендерит вопрос через `renderQuestion()`.

`renderQuestion(d)` — строит HTML вопроса: текст, варианты (кнопки с буквами), прогресс-бар (answered/total), IRT-шкала θ через `buildGauge()`.

`submitAnswer(letter)` — отправляет ответ (`POST /sessions/{id}/answer`), рендерит фидбек через `renderFeedback()`.

`renderFeedback(d)` — показывает: верно/неверно, правильный вариант, объяснение, типичную ошибку, блок заблуждения (если есть). Кнопка «Дальше» → `nextQuestion()`.

### Визуализация

`buildGauge(theta, level)` — SVG-шкала IRT от -3 до +3 с тиками, цветовым градиентом и иглой на позиции θ.

`moveGauge(theta, level)` — плавно двигает иглу после ответа через CSS-transition.

`sparkline(series)` — SVG-график θ по вопросам: линия тренда, цветные точки (зелёный = верно, красный = неверно).

### API-клиент

`api(path, opts)` — обёртка над `fetch()`. Автоматически добавляет `Content-Type: application/json`, сериализует body. При ошибке HTTP бросает `Error` с текстом из `detail` поля ответа.

### STAGE_ORDER и STAGE_HINTS

`STAGE_ORDER` определяет порядок ступеней в выпадающем списке: grade_1 → grade_11 → auto. Диапазонные псевдонимы (primary/easy/medium/hard/expert) не показываются пользователю.

`STAGE_HINTS` — подсказки под выпадающим списком для каждой ступени.
